CitrusButter

一个工程化、模块化、可测试、可扩展的 coding agent runtime kernel。 V1 已经具备真实 runtime loop、多模型 adapter、本地工具、权限审批、 session 事件流、memory 边界和 CLI 入口。

项目定位

不是复刻 Claude Code，而是重建 harness 内核。

CitrusButter 的核心目标是展示软件工程能力：将模型接入、工具执行、 本地权限、上下文组装、会话事件、memory 和未来 MCP 接入拆成稳定边界。

V1 遵循一个简单规则：AgentRuntime 拥有 loop，其他模块都是可替换依赖。 未来新增 provider、tool、memory 后端或 MCP tool source 时，应以新增实现和测试为主， 不横向修改 runtime 主流程。

Overall Architecture

整体架构设计

CLI 只是 SDK 的第一方客户端。所有核心行为都通过 runtime kernel 协调， 并通过端口接口连接到 provider、tool、permission、context、session、memory 和 observer。

Component Catalog

每个组件包含什么

V1 的组件刻意保持窄接口：每个模块只做一件事，并通过明确模型交换数据。 下面按职责解释每个部分的内容、输入、输出和扩展方式。

01

CLI Layer

`src/citrus/cli/app.py` 提供 `citrus run`、`citrus providers`、 `citrus config`。它不拥有业务逻辑，只负责解析参数、创建 provider、 组装 runtime，然后输出结果。

包含
Typer app、provider 选择、fake demo、配置展示。
交互
调用 `build_provider` 或 `FakeProvider`，再调用 `AgentRuntime.run()`。
02

Runtime Kernel

`AgentRuntime` 是系统中心。它创建 session，构建 context，调用模型， 收集 tool call，执行权限判断，运行工具，追加 tool result， 并记录结构化事件。

包含
`RunRequest`、`RunResult`、turn loop、event emit、tool execution。
交互
依赖 provider、registry、policy、context、session、memory、observer。
03

Provider Adapters

Provider 层把 CitrusButter 的内部 `Message`、`ToolSpec`、`ToolCall` 转换成各模型厂商格式。OpenAI 与 DeepSeek 使用 OpenAI-compatible chat completions；Anthropic 使用 messages API 形态。

包含
`AnthropicProvider`、`OpenAIProvider`、`DeepSeekProvider`、`FakeProvider`。
交互
接收 `ModelRequest`，返回统一 `ModelResponse`，避免厂商格式泄漏到 runtime。
04

Tool System

Tool 是 agent 能触达本地环境的唯一能力出口。V1 提供文件读取、 文件写入、文本搜索和 shell 命令，全部限制在 workspace 语义下运行。

包含
`Tool` protocol、`ToolContext`、`ToolRegistry`、`ToolSource`、local tools。
交互
runtime 根据 model tool call 从 registry 取工具，并传入 workspace context。
05

Permission Policy

权限层在工具执行前做决策。默认策略允许读文件；写文件和 shell 命令返回 ask；明显危险命令直接 deny。Runtime 通过 auto approve、 `PermissionApprover` 或安全拒绝来解析 ask。

包含
`PermissionDecision`、`PermissionRequest`、`PermissionApprover`、`GradedPermissionPolicy`。
交互
runtime 在每次 tool call 前发出 permission events，并把 ask 交给 approver 或默认拒绝。
06

Context Builder

V1 的 context deliberately small：先把用户任务转换成内部 message。 这个模块是未来检索、repo summary、memory context 和 tool result summary 的入口。

包含
`ContextBuilder.build(task)` 与内部 `Message` 输出。
交互
runtime 每次任务开始先调用它，provider 只看到统一后的 messages。
07

Session And Events

Session 不是 memory，而是一次任务的审计轨迹。事件流记录 task、 context、model request/response、tool、permission、completion 和 failure。

包含
`EventType`、`SessionEvent`、`InMemorySessionStore`、`JsonlSessionStore`。
交互
runtime 每个关键步骤 append event；observer 可以订阅这些 event。
08

Memory Boundary

V1 不实现复杂长期记忆，但稳定了 future memory 的接口位置。 Memory 作为跨任务事实系统存在，和 session history、context input 分离。

包含
`MemoryItem`、`MemoryCandidate`、`MemoryStore`、`MemoryService`、`NoopMemoryStore`。
交互
未来通过 `ContextSource` 检索，通过 `RuntimeObserver` 生成候选记忆。

Runtime Flow

组件之间如何交互

一次 `citrus run` 会穿过 CLI、runtime、context、provider、permission、 tools 和 session event stream。下面是 V1 的真实交互路径。

  1. `citrus run` 读取 provider、model、fake response 和环境变量， 通过 `build_provider` 或 `FakeProvider` 创建模型 adapter。

  2. `AgentRuntime.run()` 创建 session id，记录 `TaskStarted`， 并准备 workspace、tool registry、permission policy 和 observers。

  3. 当前 V1 把用户任务转换为 `Message.user_text()`。 这一步未来会接入 repo summary、memory retrieval 和 file snippets。

  4. Runtime 把 messages 和 ToolRegistry 导出的 `ToolSpec` 组合成 `ModelRequest`。Provider adapter 负责映射到各厂商 API。

  5. 如果模型返回 tool call，runtime 先记录 `ToolRequested`， 再调用 `PermissionPolicy`。allow 直接执行，deny 直接失败，ask 会进入 auto approve 或 `PermissionApprover` 审批；没有 approver 时默认拒绝。

  6. 工具在 workspace 内运行，返回 `ToolResult`。Runtime 把结果追加为 tool message， 进入下一轮模型调用，直到模型返回最终文本。

  7. 每个关键动作都会写入 `SessionEvent`。JSONL store 可持久化，observer 未来可用于 hooks、tracing、eval 和 memory extraction。

Extension Strategy

未来扩展如何保持轻量

V1 的重点不是把所有未来能力都做完，而是让未来能力以新增实现的方式进入系统。 这些扩展点已经由测试覆盖。

新增模型

实现 `ModelProvider`，把外部 API 映射到 `ModelRequest` / `ModelResponse`。

新增工具

实现 `Tool`，注册到 `ToolRegistry`。权限仍由 runtime 统一处理。

接入 MCP

实现 `MCPToolSource`，把远程工具转换为标准 `Tool`，无需改 runtime loop。

扩展 Memory

替换 `MemoryStore`，或用 observer 从 session events 中生成 memory candidate。

增加 Eval / Tracing

实现 `RuntimeObserver` 订阅事件流，导出 traces、metrics 或回放数据。

增强 Workspace Safety

替换 local tools 或 permission policy，引入 sandbox、worktree 或 diff review。
